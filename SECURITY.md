# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 2.x.x   | :white_check_mark: |
| 1.x.x   | :x:                |

## Reporting a Vulnerability

**We take security seriously.** If you believe you've found a security vulnerability, please follow these steps:

### DO NOT:
- Create public GitHub issues for security vulnerabilities
- Discuss the vulnerability in public forums
- Share exploit code publicly

### DO:
1. **Email us immediately** at [security@your-domain.com]
2. Include detailed information about the vulnerability
3. Provide steps to reproduce if possible
4. Allow us 72 hours to respond before any disclosure

### What to Include:
- Type of vulnerability
- Affected components/versions
- Steps to reproduce
- Potential impact
- Suggested fixes (if any)

## Response Process

1. **Acknowledgment**: We will acknowledge receipt within 72 hours
2. **Investigation**: Our security team will investigate the report
3. **Fix Development**: We'll develop and test a fix
4. **Disclosure**: We'll coordinate disclosure timing with the reporter
5. **Release**: We'll release patched versions

## Security Best Practices

### For Users:
- Keep software updated to latest versions
- Use strong, unique passwords
- Enable 2FA where available
- Follow principle of least privilege

### For Contributors:
- Never commit secrets or credentials
- Follow secure coding practices
- Use dependency scanning tools
- Participate in security reviews
